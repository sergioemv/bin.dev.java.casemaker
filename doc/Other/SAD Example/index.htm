This document presents the architectural as a series of views; use case view, process view, deployment view, and implementation view. These views are presented as Image Files (*.jpg) and use the Unified Modeling Language (UML) version 2.

 

1.6 Architectural Goals and Constraints 

There are some key requirements and system constraints that have a significant bearing on the architecture. They are:

                     The system must be done in Eclipse Platform.

                     For the representation of the Graphics in the User Screen, the system must uses GEF libraries, and Draw2d libraries.

                     All wizards and dialogs must be done in Jface, but in the case Jface don’t provide a component that the system need, only in that way the system should use SWT libraries.  

                     The system must be a RCP Application, but if this technology not yet in high level, then  the system must be a Plug-in.

                     The connection between the PD Model and the system must be trough a Plug-in, that have the goal to communicate the Old CaseMaker based in Swing, with the new made in Eclipse Platform.

                     All the Old CaseMaker must be migrated to Eclipse Platform.

2.3.1.1 VTCD Use Case - Initiate Visual Test Cases Design 

This is the first use case, this is necessary to start the application for the first time.

Basic Flow

  • The user initiates CaseMaker, creates or opens a workspace

  • The user creates a new project

  • Goes to Visual requirements node in the left tree panel

  • The user selects the Visual Test Cases Design Tab

  • The system shows the main view with no elements and ready to add them

Alternative Flows

Project already exists

  • The user initiate CaseMaker, creates  or opens a workspace

  • Opens a new projects

  • Goes to Visual Requirements node in the left tree panel

  • The user selects the Visual Test Cases Design tab

  • The system creates a default layout with the existing  elements and dependencies and ready to work with

 

2.3.1.2 VTCD Use Case - Toggles Diagram Visibility 

Basic Flow

  • User is on the main or combination diagram

  • User goes to the menu project --> Hide (or Show)

  • The diagrams tabs disappears (or appears)

 

2.3.2. Manages Elements 

The use cases that are inside the first iteration are explained below the graphic.

 

2.3.2.1 VTCD Use Case - Add Elements 

Basic Flow

  • The user clicks the menu add element to add a new element in the test mode
  • The new element dialog appears
  • The user inputs the name and description
  • Optionally the user inputs an icon path and if the element is visible in the diagram
  • The user clicks the ok button
  • The element is added in the diagram and in the grid 

Alternative Flows

The user Draws the element

  • The user clicks on the toolbar the "draw element" button

  • The user clicks on the diagram's canvas

  • The new element dialog appears

  • the user inputs the name and description (and optional data)

  • the user clicks the ok button

 

 2.3.3. Manages Equivalence Classes Diagram 

 The use cases that are inside the first iteration are explained below the graphic.

 

 

2.3.3.1 VTCD Use Case - Add Equivalence Class 

 Basic Flow

  • The user selects an element of the diagram
  • from the menu or the popup selects "Create equivalence class"
  • The create equivalence class dialog appears
  • The user fills the data and clicks ok
  • The equivalence class is added to the diagram and the grid 

Alternative Flows

User draws the equivalence class

  • The user select an element in the diagram
  • The user selects the "draws equivalence class" from the toolbar
  • The create equivalence class dialog appears
  • The user fills the input data and clicks ok
  • The new equivalence class appears

 

2.3.4. Manages Dependencies Diagram 

The use cases that are inside the first iteration are explained below the graphic.

 2.3.4.1 VTCD Use Case - Add Dependency 

 Basic Flow

  • The user drags a dependency from the toolbar to the diagram

  • The new dependency dialog appears 

  • The user fills the dependency  data and clicks ok

  • a line between the dependency and all the elements appears

  • the combination view becomes active (if the dependency is created) 

 

 2.3.5. Manages Combinations 

 The use cases that are inside the first iteration are explained below the graphic.

 

 2.3.5.1 VTCD Use Case - Add Combination 

 Basic Flow

  • User Selects two or more equivalence classes (showing a line)
  • User Clicks the menu  Combination -> Create
  • The combination dialog appears
  • The user fills the data
  • A line is drawed between the equivalence classes 

Alternative Flows

User draws the combination

  • User selects the draw combination icon from the draw toolbar

  • User selects a equivalence class and drags the line to other equivalence class

  • User clicks the draw combination icon again to commit the combination

  

2.3.6. Manages Cause – Effects 

 The use cases that are inside the first iteration are explained below the graphic.

 2.3.6.1 VTCD Use Case - Assign Cause Effect 

 Basic Flow

  • User selects a Cause effect from the cause effect tab
  • User Drags the Cause Effect to a combination or equivalence class in the diagram
  • The Cause effect is added to the combination or cause effect and displayed in the diagram

Alternative Flows

The traditional way

  • The user selects a combination or equivalence class ( in the grid or the diagram)
  • The user right clicks to the popup menu and selects "create cause effect" or "assign cause effect"
  • The Cause effects dialog appears
  • The user creates or edits the current cause effects assigned
  • The Cause effect is added (or removed) from the combination or cause effect and displayed in the diagram

4 Logical View  

4.1 Overview 

A description of the logical view of the architecture. Describes the most important classes, their organization in service packages and subsystems, and the organization of these subsystems into layers. Also describes the most important use-case realizations, for example, the dynamic aspects of the architecture. Class diagrams are included to illustrate the relationships between architecturally significant classes, subsystems, packages and layers.

This part present a tree layer model, with View, Controller and Model, and below we show a structure of the different levels of hierarchy of the subsystems.

The logical view of the CaseMaker – Visual Test Cases Design is comprised of 4 main packages: 

 

 4.2 View 

This Package is composed for all the UI and all the Graphical Elements in the systems, this includes Wizards, Dialogs, and the most important part of the system, the figures that the User can manipulate to achieve his goal.

 4.3 Controller 

 This package and the sub-package content all the logic of the CaseMaker – Visual Test Cases Design, one objective is to draw the graphics in the diagrams, and control this graphics, this package have to listen to all the actions that the user want to do, and respond to this request with the right answer, and do exactly what the user wants.

 Other Objective of this package is to request to the Plug-in Model to get and set information that the user want to view or write.

 4.4 Model 

 This is a Plug-in and his job is to connect the persistent data to the VTCD system, this plug-in answer to the getter and setter request.

 Also this plug-in validate the data before store it.

 4.5 Data 

 This is a representation of the logical view of the persistent model, in other words here are the data that the user store where he push the Save button.

 In CaseMaker – Visual Test Cases Design this is a XML file.
